AI-Agenten für SAP CRUD — drei Pfade für FSI-Kunden.

S/4HANA Private Cloud, ECC, Treasury Risk Management. Drei sehr unterschiedliche Architektur-Pfade führen zur gleichen Endauswirkung — ein Agent, der Buchungen liest und schreibt. Der Pfad bestimmt Lizenzkosten, Audit-Story, Vendor-Lock und Time-to-Production. Nicht das Frontend.

Stand Saturday, 25 April 2026
Adelshofen, BY
Die Frage
Wie verbinden wir AI-Agenten mit SAP-Backends unserer FSI-Klientel — auch on-prem & PCE — so, dass CRUD audit-fest läuft?
Drei Pfade
01
SAP-nativ

Joule + MCP Gateway

via SAP Integration Suite
Joule Studio Agent  →  MCP Gateway (IS)  →  Cloud Connector  →  S/4HANA
WannIS-Lizenz + Joule, S/4 PCE, AI-Roadmap definiert
AufwandMittel · Skill/Action pro Use Case, Tool-Spec im Gateway
IdentityPrincipal Propagation, Audit-Trail über BTP / CALM
Lock-inHoch — voll im SAP-Stack
RisikoLizenzkosten IS + Joule, Latenz durch zwei Hops
StärkeStrategisch sauberster SAP-Pfad, von SAP investiert
Best Fit Großbanken & Versicherer mit BTP-Strategie und Joule-Lizenz.
02
In-System

ABAP MCP-Add-on

MCP-Server direkt im SAP-System
Agent (Claude / Copilot / Joule)  →  SICF-Endpoint  →  ABAP-Tools auf RAP / BAPIs
WannECC oder S/4 PCE ohne IS, schlanke Architektur, kostensensitiv
AufwandNiedrig–Mittel · Add-on installieren + RAP-/BAPI-Wrapper
IdentityKlassische ABAP-Berechtigungen — kein zusätzlicher Layer
Lock-inNiedrig — Agent-Wahl frei (MCP-Standard)
RisikoSICF-Hardening Pflicht; Add-on-Support muss geklärt werden
StärkeFunktioniert ab 7.01; kürzeste Latenz; Tools beim Datenmodell
Best Fit Sparkassen, VR-Banken, Mittelstand-Versicherer ohne BTP-Investment.
03
Custom

Externer MCP-Server

Python / Node, OData & RFC zu SAP
Agent  →  Python MCP-Server (Container)  →  PyRFC / OData  →  SAP
WannPoCs, Demos, Sales-Vehikel, Wrapper für eigene IP
AufwandNiedrig (Demo) · Mittel (Prod — Idempotenz, Audit selbst)
IdentityService-User oder OAuth-Tausch, selbst zu modellieren
Lock-inSehr niedrig — Modell, Agent & Hosting frei wählbar
RisikoEigener Betriebsstrang; Compliance-Story handgeschrieben
StärkeMaximale Freiheit; reife SDKs; ECL-Engine als Demo wrapbar
Best Fit CFO-Sales-PoCs & eigener Produktstrang um die ECL Posting Engine.

iii.
Pfad 1 nur opportunistisch
Wir gehen Pfad 1 nur, wenn der Kunde Integration Suite und Joule bereits lizenziert hat. Sonst verlieren wir Zeit in Lizenzgesprächen, in denen x1F nichts mehrert.

Vier Compliance-Guardrails — sonst sagt der Compliance-Officer „nein".

01
Idempotenzschlüssel pro Aufruf
Serverseitig erzwungen, gegen Retry-Doppelpostings durch fehlgeschlagene Streams.
02
Pre-Authorization-Tool
Separates Read-only-Tool vor jedem Schreibtool, prüft Berechtigung & Limits.
03
Two-Phase-Pattern
Tool 1 berechnet Vorschlag & gibt Hash zurück. Tool 2 persistiert nach Approval.
04
Application-Log mit Agent-ID
Agent-ID + Prompt-Hash pro Posting im SLG1, abgreifbar durch Innenrevision.
